# Pre-Deployment Checklist - CCOE Insurance Industry LLC

> **Note**: This checklist is now part of the consolidated DEPLOYMENT_GUIDE.md in the root directory. Use that guide for complete deployment instructions.

## ✅ Prerequisites Verification

### 1. AWS Account Setup
- [ ] AWS Account with administrator access
- [ ] AWS CLI installed and configured
- [ ] AWS credentials configured (`aws configure`)
- [ ] Correct region set (recommended: us-east-1)

### 2. AWS Service Access
- [ ] Amazon Bedrock access enabled
- [ ] Bedrock model access granted:
  - [ ] Claude Sonnet 4 (via cross-region inference: us.anthropic.claude-sonnet-4-20250514-v1:0)
  - [ ] Titan Embeddings v2 (amazon.titan-embed-text-v2:0)
- [ ] Service quotas verified for:
  - [ ] Lambda concurrent executions (minimum: 100)
  - [ ] DynamoDB tables (minimum: 10)
  - [ ] S3 buckets (minimum: 5)

### 3. Development Tools
- [ ] Node.js 18+ installed (`node --version`)
- [ ] npm installed (`npm --version`)
- [ ] Python 3.11+ installed (`python3 --version`)
- [ ] pip installed (`pip3 --version`)
- [ ] AWS CDK CLI installed (`cdk --version`)
- [ ] Git installed (`git --version`)
- [ ] Python packages for OpenSearch index creation (`pip3 install boto3 opensearch-py requests-aws4auth`)

### 4. System Requirements
- [ ] Minimum 8GB RAM available
- [ ] Minimum 10GB disk space available
- [ ] Stable internet connection

## ⚠️ Common Issues

### Issue: Bedrock Access Denied
**Solution**: 
1. Go to AWS Console → Bedrock
2. Click "Model access" in left menu
3. Click "Manage model access"
4. Enable Claude 3.5 Sonnet, Claude 3 Haiku, and Titan Embeddings
5. Wait 5-10 minutes for access to be granted

### Issue: AWS CDK Not Found
**Solution**:
```bash
npm install -g aws-cdk
```

### Issue: Python Version Too Old
**Solution**:
- macOS: `brew install python@3.11`
- Ubuntu: `sudo apt install python3.11`
- Windows: Download from python.org

### Issue: Insufficient Permissions
**Solution**: Ensure your IAM user/role has these policies:
- AdministratorAccess (for demo), or
- Custom policy with permissions for: Lambda, DynamoDB, S3, Bedrock, Cognito, API Gateway, CloudWatch, IAM

## 🔧 What You Need to Provide vs What's Auto-Generated

When deploying this solution to your own AWS account, most resource identifiers are created automatically by CDK. Here's what you need to know:

### You Provide (before deployment)

| Item | Where | Notes |
|------|-------|-------|
| AWS Account with admin access | `aws configure` | Any account works — no specific account ID is hardcoded |
| AWS Region | `scripts/deploy.sh` | Defaults to `us-east-1`; change `REGION` variable if needed |
| Bedrock model access | AWS Console → Bedrock → Model access | Enable Claude Sonnet 4 and Titan Embeddings |

That's it. Everything else is created for you.

### Auto-Generated by CDK (outputs.json)

After running `cdk deploy --all --outputs-file outputs.json`, these values are generated and written to `backend/infrastructure/outputs.json`:

| Value | CDK Output Key | Used By |
|-------|---------------|---------|
| API Gateway URL | `LifeInsuranceApiStack.ApiUrl` | Frontend `.env` (`VITE_API_URL`) |
| Cognito User Pool ID | `LifeInsuranceInfraStack.UserPoolId` | Frontend `.env` (`VITE_USER_POOL_ID`) |
| Cognito Client ID | `LifeInsuranceInfraStack.UserPoolClientId` | Frontend `.env` (`VITE_USER_POOL_CLIENT_ID`) |
| CloudFront Distribution ID | `LifeInsuranceInfraStack.CloudFrontDistributionId` | Cache invalidation after frontend deploy |
| CloudFront URL | `LifeInsuranceInfraStack.FrontendURL` | Your live site URL |
| Frontend S3 Bucket | `LifeInsuranceInfraStack.FrontendBucketName` | `aws s3 sync dist/` target |
| Documents S3 Bucket | `LifeInsuranceInfraStack.DocumentsBucketName` | Cleanup script |
| OpenSearch Endpoint | `LifeInsuranceInfraStack.OpenSearchEndpoint` | Index creation script (auto-discovered) |
| Knowledge Base IDs | *(created internally by CDK)* | Referenced by Lambda via environment variables |
| Guardrail ID | *(created internally by CDK)* | Referenced by AgentCore runtimes |

### How the Frontend Gets Configured

The `scripts/deploy.sh` script (Phase 6) automatically:
1. Reads `outputs.json`
2. Extracts API URL, Cognito Pool ID, and Client ID
3. Writes `frontend/.env`
4. Builds and deploys the frontend

If deploying manually, copy values from `outputs.json` into `frontend/.env` using the template in `frontend/.env.example`.

> **⚠️ Important**: The `frontend/.env` file has been sanitized to remove deployment-specific values (API Gateway URL, Cognito Pool ID, Client ID). Before building or deploying the frontend, you must re-populate it with your actual values from `outputs.json`. The file currently contains placeholders. Either run `scripts/deploy.sh` (which writes `.env` automatically) or manually fill in the values from `outputs.json`:
> ```
> VITE_API_URL=https://<YOUR_API_GATEWAY_ID>.execute-api.us-east-1.amazonaws.com/prod
> VITE_USER_POOL_ID=<YOUR_COGNITO_USER_POOL_ID>
> VITE_USER_POOL_CLIENT_ID=<YOUR_COGNITO_USER_POOL_CLIENT_ID>
> ```
